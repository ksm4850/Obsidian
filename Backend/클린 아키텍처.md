# 클린 아키텍처 (Clean Architecture)

Robert C. Martin(Uncle Bob)이 제안한 소프트웨어 설계 원칙. 핵심은 **의존성 방향이 항상 안쪽(비즈니스 규칙)을 향해야 한다**는 것.

## 핵심 원칙

| 원칙 | 설명 |
|---|---|
| **의존성 규칙** | 바깥 계층이 안쪽 계층에 의존. 안쪽은 바깥을 모름 |
| **프레임워크 독립성** | FastAPI, Django 등 프레임워크는 교체 가능한 도구일 뿐 |
| **테스트 용이성** | 비즈니스 로직을 DB, 웹 없이 단독 테스트 가능 |
| **UI/DB 독립성** | UI나 DB를 교체해도 비즈니스 로직은 변경 없음 |

## 4개 계층 (안쪽 → 바깥쪽)

### 원서 다이어그램 (Uncle Bob 원본)

```
┌─────────────────────────────────────────┐
│        Frameworks & Drivers             │  ← FastAPI·SQLAlchemy 라이브러리 자체
│  ┌───────────────────────────────────┐  │
│  │        Interface Adapters         │  │  ← Controller, Repository 구현, Gateway
│  │  ┌─────────────────────────────┐  │  │
│  │  │      Use Cases              │  │  │  ← 비즈니스 흐름 조합
│  │  │  ┌───────────────────────┐  │  │  │
│  │  │  │      Entity            │  │  │  │  ← 핵심 비즈니스 규칙
│  │  │  └───────────────────────┘  │  │  │
│  │  └─────────────────────────────┘  │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

원서의 **Frameworks & Drivers**(가장 바깥)는 FastAPI·SQLAlchemy **라이브러리 자체**를 의미한다. pip 의존성일 뿐 폴더로 관리하지 않는다.

원서의 **Interface Adapters**에는 웹 쪽(Controller, Presenter)과 DB 쪽(Repository 구현, Gateway) **모두** 포함된다.

### 실무 폴더 구조 (4-폴더)

실무에서는 Interface Adapters를 **방향별로 분리**한다:

```
┌───────────────────────────────────────────────────┐
│  presentation/     │  infrastructure/             │  ← 둘 다 Interface Adapters
│  (웹 쪽 어댑터)     │  (DB·외부 서비스 쪽 어댑터)    │
│  ┌─────────────────────────────────────────────┐  │
│  │              application/                   │  │  ← Use Cases
│  │  ┌───────────────────────────────────────┐  │  │
│  │  │            domain/                    │  │  │  ← Entity
│  │  └───────────────────────────────────────┘  │  │
│  └─────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────┘
```

> **핵심**: `presentation/`과 `infrastructure/`는 같은 계층(Interface Adapters)을 **역할별로 쪼갠 것**이다. 상하 관계가 아니라 **동일 레벨**.

### 원서 → 실무 용어 매핑

| 원서 (Uncle Bob) | 실무 폴더 | 설명 |
|---|---|---|
| Entity | `domain/` | 핵심 비즈니스 규칙 |
| Use Cases | `application/` | 비즈니스 흐름 조합 |
| Interface Adapters (웹 쪽) | `presentation/` | 라우터, 스키마, 컨트롤러 |
| Interface Adapters (DB 쪽) | `infrastructure/` | Repository 구현체, 외부 서비스 연동 |
| Frameworks & Drivers | 폴더 없음 | pip 의존성 (FastAPI, SQLAlchemy 등) |

### 1. domain (Entity) — 가장 안쪽

- 비즈니스 핵심 규칙과 엔티티
- 어떤 외부 라이브러리에도 의존하지 않음
- 예: `User`, `Order` 같은 도메인 모델과 비즈니스 규칙

### 2. application (Use Cases)

- 애플리케이션 고유의 비즈니스 흐름 조합
- Domain 계층만 의존
- 예: "주문 생성", "유저 등록" 같은 시나리오

### 3. presentation (Interface Adapters — 웹 쪽)

- 외부 HTTP 요청을 내부 Use Case 형식으로 변환, 응답을 외부 형식으로 변환
- 예: FastAPI 라우터, Pydantic Request/Response 스키마, DI 설정

### 4. infrastructure (Interface Adapters — DB·외부 서비스 쪽)

- application에서 정의한 Repository 인터페이스의 **구현체**
- 예: SQLAlchemy ORM 모델/구현체, Redis 클라이언트, 이메일 서비스

## 의존성 방향

```
Router → Use Case → Domain
  ↓
Repository(구현) → Repository(인터페이스) ← Use Case
```

Use Case는 Repository **인터페이스**(추상)에 의존하고, 실제 DB 구현체는 그 인터페이스를 구현. 이것이 **의존성 역전(DIP)**.

## FastAPI 폴더 구조

```
project/
├── app/
│   ├── main.py                     # FastAPI 앱 생성, 라우터 등록
│   ├── domain/                     # 계층 1: 도메인
│   │   ├── entities/
│   │   │   ├── user.py             # 도메인 엔티티 (순수 Python 클래스)
│   │   │   └── order.py
│   │   ├── value_objects/
│   │   │   ├── email.py            # 값 객체 (불변, 자체 검증)
│   │   │   └── money.py
│   │   └── exceptions.py           # 도메인 예외
│   │
│   ├── application/                # 계층 2: 유스케이스
│   │   ├── use_cases/
│   │   │   ├── create_user.py      # 유스케이스 (비즈니스 흐름)
│   │   │   └── create_order.py
│   │   ├── interfaces/             # Port (추상 Repository)
│   │   │   ├── user_repository.py  # ABC / Protocol
│   │   │   └── order_repository.py
│   │   └── dto/
│   │       ├── user_dto.py         # 계층 간 데이터 전달 객체
│   │       └── order_dto.py
│   │
│   ├── infrastructure/             # Interface Adapters (DB·외부 서비스 쪽)
│   │   ├── database/
│   │   │   ├── connection.py       # DB 세션 관리
│   │   │   └── models.py           # SQLAlchemy ORM 모델
│   │   ├── repositories/           # Adapter (Repository 구현체)
│   │   │   ├── user_repository.py  # 인터페이스 구현
│   │   │   └── order_repository.py
│   │   └── external/
│   │       └── email_service.py    # 외부 API 연동
│   │
│   └── presentation/               # Interface Adapters (웹 쪽)
│       ├── api/
│       │   └── v1/
│       │       ├── user_router.py  # FastAPI 라우터
│       │       └── order_router.py
│       ├── schemas/
│       │   ├── user_schema.py      # Pydantic Request/Response
│       │   └── order_schema.py
│       └── dependencies.py         # FastAPI Depends (DI 연결)
│
├── tests/
│   ├── unit/                       # 도메인/유스케이스 단위 테스트
│   └── integration/                # DB 연동 통합 테스트
└── pyproject.toml
```

## 계층별 의존 관계

| 계층 | 의존 대상 | 절대 의존 금지 |
|---|---|---|
| `domain/` | 없음 (순수 Python) | 모든 외부 계층 |
| `application/` | `domain/` | `infrastructure/`, `presentation/` |
| `infrastructure/` | `application/interfaces/`, `domain/` | `presentation/` |
| `presentation/` | `application/`, `domain/` | `infrastructure/` 직접 참조 (DI로 주입) |

## 자주 혼동되는 포인트

- **Entity ≠ ORM 모델**: `domain/entities/`는 순수 Python 클래스, `infrastructure/database/models.py`는 [[Sqlalchemy]] 모델. 둘은 별개
- **Repository 인터페이스는 application에**, 구현체는 infrastructure에 위치
- **FastAPI의 `Depends`**가 DI 컨테이너 역할 → `dependencies.py`에서 구현체를 인터페이스에 바인딩
